# architecture/

System structure — how the project is built.

## What belongs here

Factual descriptions of the system's components, data model, and design patterns. One subsystem or concern per file.

**Examples:**
- `module-inventory.md` — what each module does and its dependencies
- `database-schema.md` — tables, primary keys, key relationships
- `data-flow.md` — how data moves through the system end-to-end
- `design-patterns.md` — patterns used and the reasoning behind them

## INDEX.md

Answers "how is this project structured?" in one screen. Typical contents:

- Code layout — top-level directories and what lives in each
- The primary data model — key entities and how they relate
- The set of modules/components a new contributor needs to know, one line each
- Links to deeper docs (schema references, ADRs, design notes) rather than inlining them

Generated by `session-start` on first run (not by hand) — see [../README.md](../README.md) for details.

## Format guidance

- Factual and current — a stale architecture doc is worse than none
- Avoid implementation detail; focus on structure and relationships
- Keep under 200 lines — if a schema is large, document the key tables and link to the migration files for the rest
