# business-rules/

Domain logic Claude needs to apply correctly.

## What belongs here

The rules of the business that aren't obvious from the code — calculations, state machines, constraints, and invariants. One domain area per file.

**Examples:**
- `calculation-rules.md` — formulas, rates, rounding behavior, edge cases
- `state-machine.md` — valid states, allowed transitions, terminal states
- `constraint-rules.md` — business invariants that must always hold
- `workflow-rules.md` — what triggers what, sequencing requirements

## INDEX.md

Answers "what domain rules govern this project?" in one screen. Typical contents:

- The core domain concepts — entities, lifecycle states, key calculations
- Pointers to the rule files in this folder that cover each concept
- Foundational rules every contributor must internalize (e.g. "all amounts are stored in cents, never dollars")
- Links to authoritative sources — legal docs, specs, regulatory references — rather than restating them

Generated by `session-start` on first run (not by hand) — see [../README.md](../README.md) for details.

## Format guidance

- Precise over readable — Claude needs to apply these rules correctly, not just understand them generally
- State the rule explicitly, then any exceptions
- Include the edge cases that trip people up
- Keep under 200 lines — one domain area per file
