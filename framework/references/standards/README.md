# standards/

Code and documentation conventions — how to write the code.

## What belongs here

Rules and patterns Claude should follow when writing or reviewing code. One domain per file.

**Examples:**
- `sql-conventions.md` — naming rules, constraint patterns, migration format
- `python-conventions.md` — type hints, error handling, module size limits
- `testing-standards.md` — test structure, naming, coverage expectations
- `api-conventions.md` — endpoint naming, request/response shape, error format

## INDEX.md

Answers "what conventions does this project follow?" in one screen. Typical contents:

- Languages and frameworks in active use and which convention file applies to each
- Project-wide rules that cut across languages (naming, doc-comment policy, error handling philosophy)
- Tooling that enforces conventions (linters, formatters, type checkers) and where their config lives
- Anything the project does *differently* from common community conventions, and why

Generated by `pre-work` or `pre-phase` on first run, not by hand.

## Format guidance

- Rule-based: state the rule, then optionally show a short example
- Explain the why only when it's non-obvious
- Keep under 200 lines — one language or concern per file
