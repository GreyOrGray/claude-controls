# workflow/

Process documentation — how to do the work.

## What belongs here

Day-to-day procedures Claude needs to follow or reference during implementation. One process per file.

**Examples:**
- `promotion-process.md` — how code moves from dev to production
- `approval-gates.md` — what requires sign-off and from whom
- `environment-setup.md` — how to create and name test environments
- `commit-workflow.md` — what to check and stage before committing

## INDEX.md

Answers "how does work happen in this project?" in one screen. Typical contents:

- The promotion path — where new code starts, how it's reviewed, how it reaches production
- The approval gates — what requires explicit sign-off and from whom
- The environment model — dev/test/staging/prod, how they're created and named
- Links to the individual process files in this folder, with a one-line description of when each applies

Generated by `session-start` on first run (not by hand) — see [../README.md](../README.md) for details.

## Format guidance

- Step-by-step or checklist style
- State the trigger: when does this process apply?
- State the outcome: what does done look like?
- Keep under 200 lines — if longer, split into separate files
